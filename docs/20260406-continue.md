# Continuation Notes -- 2026-04-06

## Current State

The SNOBOL4 interpreter is being split from a 54KB monolith
(src/snobol4.plsw) into 4 modular .plsw files that compile
separately and link via assembly concatenation.

### Modules

```
Layer 1: src/sno_util.plsw  -- string helpers, I/O, PUT_DEC, FIND_SPECIAL
Layer 2: src/sno_lex.plsw   -- lexer, parser, symbol/label tables
Layer 3: src/sno_exec.plsw  -- AM emitter, lowering, executor, patterns
Layer 4: src/sno_main.plsw  -- MAIN entry point (thin)
```

Shared definitions: include/snoglob.msw (%DEFINE constants + DCL globals)

### Build Pipeline

```
scripts/build-modular.sh:
  1. Compile each .plsw separately -> .s assembly
  2. Strip runtime preamble from library modules (strip-runtime.sh)
  3. Prefix internal labels (L0-L999) with module name
  4. Concatenate: main.s + util.s + lex.s + exec.s -> snobol4.s
  5. Assemble and run
```

### What Works

- All 4 modules compile individually
- strip-runtime.sh removes _start, UART_PUTCHAR/PUTS, dummy MAIN
- Label prefixing avoids L-label collisions between modules
- Concatenation produces valid assembly (no assembly errors when
  data sections are kept)

### What's Broken

**Duplicate .data sections**: Each module includes snoglob.msw which
declares shared globals (DCL SRC, SB, VARS, etc.). Each module's
assembly emits its own `.data` section with these labels. When
concatenated, all 4 copies of `_SRC:`, `_SB:`, `_VARS:` etc. end
up in the binary at different addresses. The assembler uses the first
definition for label resolution, but the later data sections waste
space and may overlap with code.

**Naive fix (stripping all .data)** breaks because library modules
also have local string literals in their .data section that they need.

### Proposed Solutions

**Option A: Smart .data stripping**
The strip script identifies which labels are "shared globals" (from
snoglob.msw) and removes only those from library modules, keeping
module-local strings (like N_OUT, N_INP) and literal data.

This requires a list of shared global labels to filter.

**Option B: EXTERNAL declarations**
PL/SW's EXTERNAL storage class should prevent emitting data for
globals declared in snoglob.msw. But testing showed EXTERNAL still
emits data labels. This needs a PL/SW compiler fix.

**Option C: Separate snoglob into %DEFINE-only and DCL-only parts**
- snoglob_defs.msw: only %DEFINE constants (no DCL)
- snoglob_data.msw: only DCL globals
Library modules include snoglob_defs.msw + use EXTERNAL for globals.
Main module includes both.

**Option D: Post-link deduplication**
After concatenation, run a script that removes duplicate label
definitions, keeping only the first occurrence of each.

**Recommended: Option A or D** -- they work with existing PL/SW.
