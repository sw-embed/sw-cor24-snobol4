Thought for 26s
You’re asking two things:

How SNOBOL4 was actually implemented historically

What the best implementation approach would be now

Historical implementation
High-level answer
SNOBOL4 was not originally bootstrapped in itself. The classic Bell Labs implementation was built as a macro-assembly implementation around a machine-independent intermediate/macro layer called SIL — the SNOBOL Implementation Language. The early SNOBOL family started in assembly on IBM mainframes, and SNOBOL4 kept that low-level implementation style, but made it more portable by expressing the interpreter in SIL and then mapping SIL onto each target machine. 

Evolution by version
SNOBOL1 was written in assembly language for the IBM 7090. 

SNOBOL3 was a rewritten and extended version, but portability problems and dialect drift pushed the designers toward a more systematic implementation strategy. 

SNOBOL4 was designed around portability, and its implementation was written in macro assembly using SIL, rather than as a self-hosting compiler. The first SNOBOL4 implementation was started on an IBM 7094 and completed on an IBM 360 in 1967. 

What SIL was
SIL was essentially a machine-independent abstract implementation language defined using macro-assembler techniques. The idea was:

write the SNOBOL4 interpreter in SIL,

implement SIL operations for each target machine,

then assemble the same core implementation on different architectures.

That gave them something very much like a portable virtual machine / abstract machine, but realized through macro assembly instead of a modern bytecode VM. Griswold’s implementation notes explicitly describe SNOBOL4 as implemented in macro-assembly language called SIL. 

Was it self-hosting?
Not in the usual sense.
The classic implementation was not “SNOBOL4 written in SNOBOL4.” It was primarily an interpreter implementation written below the language level, in SIL/macro assembly. Later systems existed:

SPITBOL: a compiler-oriented/high-performance implementation

SITBOL: a native PDP-10 implementation in assembler

CSNOBOL4: a modern C port of the macro/SIL implementation 

So if by “bootstrapped” you mean “compiler/interpreter written in its own language,” the answer is generally no for the original SNOBOL4 line.

What this means architecturally
SNOBOL4’s implementation problem is unusual. It is not “just parse expressions and emit code.” It has some features that strongly shape the implementation:

first-class patterns

backtracking pattern matching

dynamic typing

garbage-collected heap

runtime code generation / on-the-fly compilation in some implementations

success/failure-driven control semantics 

That is why historically it was commonly implemented as an interpreter/abstract machine, not as a straightforward native compiler.

The key historical fact that matters for your design
The original portable SNOBOL4 line was implemented in SIL, a macro-assembly language for an abstract machine, not for a specific hardware ISA. Bell/Arizona documentation says SIL was intended to be implemented on each target by providing target-machine macro definitions, and that “nearly all the logic” of SNOBOL4 lived in the SIL program itself. The transporter then adapted data layouts, syntax tables, and about 130 SIL macros to the target. The same notes also say the classic System/360/370 implementations needed a fairly large single image and were commonly moved by translating the SIL source and then supplying target-specific macros and subroutines. 

Why your PL/SW is a particularly good fit
From what you described, PL/SW already wants to be:

That is almost exactly the job SIL was built to do. Historically, SIL did not assume much about the target machine; one transport paper explicitly says it does not even require hardware registers, though target word size and descriptor fit matter a lot for efficiency.

3. Treat descriptors as the center of the system

That matters a lot for you:

On COR24

Stack strategy
Your fast stack is only 3–8K. That is too precious to use casually.

Rule: the fast stack is for execution frames, not language-state bulk
Use the fast stack for:

Do not use it for:

rollback history

deep pattern backtracking state

variable binding logs

dynamic strings

Those should go in the heap.

Use explicit heap-backed stacks
You should define at least three logical stacks, probably heap-backed:

1. Evaluation stack
For intermediate expression results if needed.

2. Backtracking stack
For pattern continuation frames:

current subject cursor

next pattern pc/node

previous bindings checkpoint

current environment snapshot id

3. Binding rollback stack
For speculative assignments/captures that must be reverted on failure.

This is extremely important because SNOBOL-style pattern execution is naturally continuation-heavy and can explode stack usage if mapped too literally to machine recursion.

Recommendation
Avoid deep recursive implementation in machine calls.

Implement pattern matching mostly as:

explicit state machine

explicit continuation frames

explicit heap-backed backtrack stack

You can still use recursive macros to generate this structure, but the generated runtime flow should be mostly iterative.

Register strategy on COR24
With only 3 GPRs, assume constant register starvation.

Suggested register discipline
Treat registers as specialized:

r0: primary scratch / return / compare

r1: secondary scratch / pointer

r2: tertiary scratch / current frame / helper

fp: frame anchor

sp: machine stack

iv/ir: per ISA role

The key is to avoid a style where every macro assumes it owns all registers.

Macro convention
Every AM macro should declare its clobber contract conceptually, even if only in comments/docs.

Example categories:

Pattern engine design
This is the hardest part. Design it first-class, not as an afterthought.

Recommended internal form
Do not execute patterns as arbitrary object trees forever.

Compile patterns to one of these:

Choice 1 — pattern graph
Nodes with explicit next/alt links.

Choice 2 — pattern bytecode
Compact op stream with jumps and frame operations.

For your environment, I slightly prefer:

Why
easier to inspect in dumps

easier to visualize

easier to debug during bring-up

recursive macro generation can target symbolic node builders nicely

Each node can represent:

failure terminal

Then the executor uses explicit backtrack frames to walk the graph.

Key invariant
All speculative effects must be rollback-able.

That means pattern operations that can assign or capture should log effects in a rollback structure before commit.

Memory management
With 1M heap, you have room, but SNOBOL4 semantics can fragment memory badly.

Recommendation: start with mark-sweep, maybe with compaction later

simplest to reason about

good for debugging object lifetimes

less descriptor rewriting complexity initially

Costs
fragmentation

maybe poor locality

Given your strong tooling appetite, start with a debuggable mark-sweep collector first.

Inline COR24 assembler use
You said PL/SW supports inline COR24 assembler. Good. Use it surgically.

Best uses for inline assembler
1. Tiny memory primitives
copy 1 descriptor

scan for terminators / character classes

2. Frame linkage helpers
prologue/epilogue patterns

multiword move helpers

3. Hot dispatch helpers
If the AM interpreter loop becomes a hotspot, inline assembly can tighten:

primitive comparisons

4. Trap/assert/debug hooks
Useful for breakpoints and emulator integration.

Do not inline assembler for semantic logic
Do not bury SNOBOL semantics inside ad hoc assembler islands. That will ruin portability and readability.

Keep semantic decisions in PL/SW + macros; keep assembler for primitive mechanics.

Portability toward later S/370-ish
Even though it is later, the design should prepare for it now.

Portability rules to enforce immediately
1. No direct assumptions that “pointer == integer”
Allow explicit conversions only.

2. No baked-in character set assumptions
Centralize all character classes and scanner tables.

3. No baked-in descriptor bit layout outside one layer
Only machine abstraction macros may know the physical layout.

4. Separate machine stack from semantic stacks
Very important for later retargeting.

5. Every heap object type must have explicit scanning logic
Needed for GC and debug tools across targets.

If you keep these boundaries, a later 24-bit S/370-ish port will be a real retarget, not a rewrite.

Biggest risks and mitigations
Risk 1 — macro tower becomes unreadable
Mitigation
Require every macro layer to have:

small examples

stable naming conventions

Keep “semantic macros” separate from “machine macros.”

Risk 2 — register pressure makes generated code terrible
Mitigation
Design macros around memory-resident temporaries and explicit scratch slots rather than pretending registers are abundant.

Risk 3 — pattern engine consumes stack or heap unpredictably
Mitigation
Use explicit heap-backed continuation frames and build tracing early.

Risk 4 — GC and rollback semantics interfere
Mitigation
Treat rollback logs and backtrack frames as first-class GC roots from day one.

Risk 5 — parser/runtime get tangled
Mitigation
Commit early to the AM boundary.
